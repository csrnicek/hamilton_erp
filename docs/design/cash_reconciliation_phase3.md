# Cash Reconciliation — Phase 3 Design Intent

**Status:** Design intent — not implementation spec
**Phase:** 3 (post-Phase-2 cart, post-Phase-2 card integration)
**Authored:** 2026-05-01
**Source:** synthesis of Chris's Phase 3 directives, captured before implementation begins
**Implementation status today:** placeholder controller in `hamilton_erp/hamilton_erp/doctype/cash_reconciliation/cash_reconciliation.py` (`system_expected = flt(0)` hardcoded; `variance_amount` field declared but never populated; no notification or workflow logic; `resolution_status` set by hand only)

---

## Why this document exists

Cash reconciliation is the most theft-sensitive surface in Hamilton's ops. The Phase 1 controller is a stub — it satisfies the schema but not the contract. The contract is everything: blind-by-design, three-number cross-check, no mid-process information leaks, venue-portable, tax-correct.

This document captures **the reasoning behind each decision**, not just the rules. Anyone implementing Phase 3 needs to understand *why* the operator never sees a dollar amount during recount attempts, *why* the Final Cash Amount (not POS-expected) flows to the GL, and *why* float handling must be venue-configurable rather than baked in. Misapplying any of these mechanically — without the reasoning — will produce a system that LOOKS like Phase 1 but silently breaks the trust model that justifies blind cash control in the first place.

If a future implementer thinks "this reveal-after-3 rule is overkill, let me simplify" — read §1 again before deleting it. The reveal-after-3 rule exists because every alternative leaks information.

## 1. Blind drop philosophy

### The decision

| Layer | Who sees what |
|---|---|
| Operator at drop time | Their own count only. **No POS expected. No prior-drop totals. No running shift total.** |
| Operator after drop | Nothing — the cash drop record is sealed; the operator can't reopen it to see how much they put in vs how much was expected. |
| Manager at recount time | The cash they physically count. **No POS expected. No operator-declared figure.** Until reveal mode, mismatch feedback is binary only ("doesn't match, recount"). |
| Manager after 3rd failed attempt | All three numbers (A=POS expected, B=Operator declared, C=Manager counted), plus the variance categorization. Reveal is irreversible; the audit log records that reveal happened, against both operator and manager identities. |
| GL / deposits / sales tax / income tax | The **Final Cash Amount** entered by the manager after reveal — never POS expected, never operator declared, never the un-resolved manager count if reveal happened. |

### Why blind

Three-number triangulation is the theft-detection model. Each of the three numbers comes from a different source, generated under different conditions:

- **A (POS expected)** = sum of cash payment lines on submitted Sales Invoices for the shift period. System-generated, immutable post-submit.
- **B (Operator declared)** = the operator's own statement of what they put in the envelope. Human-generated under conditions where the operator could be skimming.
- **C (Manager counted)** = the manager's physical count. Human-generated by a different person, under conditions where the manager could be colluding with the operator OR could be honest.

If any single party can see any other party's figure during the count, triangulation collapses. The three independent measurements stop being independent. Specifically:

- **If the operator sees POS expected before declaring**, they can short the till by a known-safe amount that won't flag (within tolerance), then declare the rounded-down number that matches expected. This is the easiest skim path and the original reason DEC-005 mandated blind cash drop for operators (Hamilton Operator role has zero permission on `POS Closing Entry` — see `_block_pos_closing_for_operator()` cited in `docs/review_package.md` DEC-005, `docs/RUNBOOK.md` §7.1, and `docs/lessons_learned.md` LL on Task 25 item 7).
- **If the manager sees operator-declared before counting**, the manager will (consciously or not) reach the operator's number by miscounting in the operator's favor. This is the human "anchoring" failure mode — even an honest manager will subconsciously round toward a stated target. Blind manager count breaks the anchoring path.
- **If the manager sees POS expected before counting**, same anchoring problem.

The only configuration that preserves triangulation is: **manager counts cold; system shows nothing until C is committed; mismatch feedback during recount is binary (no $ amount, no direction)**. Binary feedback is the hardest constraint — it's tempting to show "off by $X" to "help" the manager find their mistake, but that single piece of information is exactly what the skim path needs to navigate around.

Reference for the underlying decision: DEC-021 (blind cash control = DocType permissions + Report/Page permissions + field masking — `docs/review_synthesis.md` line 59). DEC-039 establishes that variance is revealed only after manager submits the blind count (`docs/field_masking_audit.md` line 97). DEC-038 establishes that the same blind-count workflow applies to card reconciliation (`docs/field_masking_audit.md` line 188 — currently violated by `Shift Record.system_expected_card_total` being readable by Operator pre-submit; that's a separate gap).

### Recount attempt mechanics

1. Manager enters count → submit.
2. System checks A ≈ B ≈ C within tolerance (`_within_tolerance` in current controller: 2% of larger amount, $1 minimum floor — preserved from Phase 1, may need to drop to $0.05 for tighter envelopes; venue-configurable).
3. If all three match: Clean. Done.
4. If mismatch: system says **"Doesn't match. Recount."** Nothing else. No amount, no direction, no hint about which of A/B/C is the outlier.
5. Manager recounts the physical cash. Resubmits.
6. Cycle repeats up to 3 attempts.
7. After the 3rd failed attempt: **reveal mode**. System discloses A, B, C, the variance amount, and the variance categorization (Possible Theft / Operator Mis-declared per the existing three-way rule in `_set_variance_flag`). Manager picks a resolution from the dropdown, writes a comment, and enters the Final Cash Amount.

The 3-attempt ceiling is a deliberate fairness floor. A skilled counter can miscount once. Twice is unusual but not damning. Three times says either the cash is genuinely wrong OR the manager's process needs intervention — either way, the human-in-the-loop path takes over and the audit log captures both the manager's repeated failures AND the operator's declaration discrepancy. Both names land in the audit trail. This protects honest managers (who get their three free shots before reveal) and catches dishonest managers (who can't iterate to "find" a number that exonerates an accomplice operator — they get one reveal per drop, period).

### Final Cash Amount is the GL truth — not POS expected

This is the subtle decision and the most important one for tax correctness.

When reveal happens and the three numbers don't agree, **someone has to pick the number that flows downstream into accounting**. The four candidates:

| Candidate | Why it's wrong as GL truth |
|---|---|
| **POS expected (A)** | Includes phantom transactions. If the POS recorded a $50 cash sale but no $50 cash exists in the envelope, the cash literally isn't there. Posting $50 to GL inflates revenue and overpays sales/income tax on money the venue doesn't have. CRA / IRS audit risk: a CRA review that compares declared revenue vs deposited cash will flag a sustained gap. |
| **Operator declared (B)** | The operator's own statement is the figure most likely to be wrong (whether by error or skim). Posting B to GL trusts the suspect party. |
| **Manager counted (C)** | The physical count is the physically correct figure for THIS envelope, but if reveal happened it means C disagreed with A and B for three attempts — there's some unresolved cause (a void that didn't post, a refund that didn't settle, an unrecorded card-tip-from-till, a genuine theft). Posting C without resolution context loses the why. |
| **Final Cash Amount** | Manager-entered after reviewing all three numbers + the resolution dropdown choice + the comment. **This is the only candidate that is both grounded in physical reality AND contextually justified.** |

Final Cash Amount is what flows to:
- **GL cash account debit** (the cash that actually exists)
- **Bank deposit slip** (the envelope's contents go to deposit)
- **Sales tax remittance** (taxable revenue is recomputed against the Final Cash Amount, not the POS-expected total — under-deposit means an offsetting adjustment line, the tax authority is paid on actual receipts not theoretical receipts)
- **Income tax** (same — actual revenue, not phantom revenue)

If POS expected was used, the venue would over-remit tax on every reconciliation that revealed a cash short. Over time this compounds into substantial real money paid to CRA / IRS for cash the venue never had. Cumulative over-remit is hard to recover on subsequent returns and effectively functions as a tax on the shrinkage itself.

The variance amount (A − Final Cash Amount) becomes a write-down posted to a designated GL account: "Cash variance — investigation," or "Cash short and over." That account is reviewed by the manager + accountant during the period close. Sustained large entries to it are a signal that something structural is wrong (a process issue, a register issue, or theft).

### Critical: operator never writes on the envelope

The operator types the count, the system prints the label, the operator affixes it. The operator never writes a number on the envelope by hand.

Why: handwritten numbers can be doctored. Pre-printed labels with the system's record of the declared amount cannot. If the operator declared $217.30 and the printed label says $217.30, the manager opens an envelope containing $180 and the system says $217.30, the discrepancy is unambiguous. If the operator handwrote "approx 217" or scratched out a wrong number, the audit becomes a forensics exercise. This is a small process discipline that prevents a nontrivial class of dispute.

Implementation note: today the print pipeline is **unbuilt** (see this morning's audit on cash drop envelope label). The Cash Drop submit hook needs to invoke a Brother QL-820NWB print (per DEC-011 in `docs/review_package.md`, formalized further in `docs/design/pos_hardware_spec.md` §5 once PR #104 merges) with the 8 spec fields from `docs/hamilton_erp_build_specification.md` §7.4 step 4. Until that pipeline lands, the blind-write discipline degrades to a procedural rule (operator writes nothing manually, manager identifies envelopes by Cash Drop record listed in Frappe).

## 2. Card tips paid from till

### The problem

Card tips are owed to the operator. The processor settles tips T+1 or T+2 to the venue's bank account, then the venue cuts a paystub. But operators want their tips in cash at end of shift — that's a near-universal industry expectation, not a Hamilton quirk.

The naive fix: operator pulls the cash equivalent of their card tips out of the till before the cash drop. The operator's pocket gets the tips. The till runs short by exactly that amount. The drop reconciles short. The system flags it as theft.

That's the failure mode. The drop doesn't know about the tip pull; it sees only that cash is missing.

### The decision

**Operator types the exact tip amount into the system. The system applies the venue's rounding rule. The system tells the operator the exact cash to take. The operator obeys.**

Concretely, in Canada:
1. Operator types `$12.67` (the card tip earned during shift, summed from terminal batch reports).
2. System applies venue rounding rule (DEC: Canada nickel rounding, round up to $0.05): result `$12.70`.
3. System displays "Take $12.70 from drawer."
4. Operator obeys. Drawer now down $12.70 as a tip-pull.
5. Cash Drop record stores the tip pull as a line item: `tip_pull_amount = 12.70`, `tip_pull_currency = CAD`, `tip_settled_via_processor_amount = 12.67` (the actual processor tip), `tip_pull_difference = 0.03` (rounding-up loss to the venue, paid as part of operating cost).
6. POS expected for reconciliation is recomputed: `system_expected_cash = sum_of_cash_sales − tip_pull_amount`. The system "knows" the till is short by $12.70 because it told the operator to take $12.70.
7. Manager reconciliation runs against the adjusted figure. No false theft flag.

### Why the system enforces the cash amount

Because operators round in their own favor when given the chance:
- "Tip is $12.67, I'll just take $13."
- "Tip is $12.67, I'll take $12.50 and split the change with my buddy."
- "Tip is $12.67, I'll take $14 because I had a bad shift."

Each of these creates a discrepancy. None are necessarily theft — many are honest "close enough" rounding by operators who are bad at math under fatigue. But the discrepancies aggregate into a pattern that reconciliation can't distinguish from skimming.

The fix is to remove the operator's discretion. The system computes the rounded amount, displays it, and the cash drop auditor (manager + the system itself) holds the operator to that exact figure. Any cash discrepancy at reconciliation is now real (not rounding noise) and warrants investigation.

### Why tip cash is NOT venue revenue

This is the tax-correctness point. The tip cash:
- Is owed to the operator, not the venue
- Is settled to the operator via processor settlement (not via the till)
- Should NOT appear in venue revenue for sales tax / income tax purposes

The till is a temporary lending mechanism — the venue is fronting the operator their own money out of the till, to be repaid when processor settlement clears. Accounting treatment: the tip pull is a debit to a "Tips Payable to Staff" liability account (or "Operator advances") and a credit to Cash. When processor settlement deposits the matching amount T+1, a corresponding entry zeros the liability against incoming bank cash.

Final Cash Amount must NOT include tip pulls as venue revenue. Sales tax remittance must NOT compute against the till total before tip pull deduction. Phase 3 implementation: the `system_expected` calculator subtracts tip pull amount; Final Cash Amount fields and the bank deposit downstream both reflect the tip-pull-adjusted figure.

The $0.03 rounding loss in the example is venue cost (an operating expense), not staff tip. Annualized across thousands of shifts, this is meaningful but small ($0.03 × 4 daily shifts × 365 = ~$44/year per venue) and the cost is justified by the consistency benefit. Track it in a "Tip Rounding Adjustments" account for transparency.

## 3. Venue-specific variations (do not hardcode)

### The decision

Every venue has its own cash-handling tradition that has accreted from local practice, local manager preferences, local regulatory environment, and local theft-experience-driven adjustments. Phase 3 must NOT hardcode Hamilton's choices as canonical. Specifically:

| Variation | Hamilton's current practice | Other venues might do |
|---|---|---|
| **Float handling at end of shift** | Subtract before drop (drop = total minus float; float stays in safe for next shift) | Don't subtract (drop = total; float is a separate envelope) |
| **Required paperwork checklist** | Card terminal batch report + signed credit card slips | Plus tax-exempt sale forms, plus comp slips, plus rental waiver acknowledgments (Philadelphia-specific), plus jurisdiction's specific receipt-retention requirements |
| **Cash drop frequency** | Mid-shift + end-of-shift (2 drops per shift) | DC at peak: 4-6 drops per shift to reduce till exposure during high-volume hours; quiet venues: end-of-shift only |
| **Reconciliation cadence** | Daily (manager arrives, opens envelopes from prior shift) | Weekly close in low-volume venues; multi-times-daily close in high-traffic venues |
| **Tolerance threshold** | 2% of larger amount with $1 floor (Phase 1) | $0.05 floor for tighter venues; per-venue regulatory or insurance-driven thresholds |
| **Tip rounding rule** | Canada: round up to $0.05 | US: round to $0.01 (no nickel rounding); some states: round to $0.05 voluntarily; jurisdictions with no cents: round to $1 |
| **Reveal threshold** | After 3rd failed recount attempt | Possibly 4 or 2 in different venues based on manager experience and operator turnover |

### Why venue-configurable, not assumed canonical

Three reasons:

1. **Regulatory variation.** US states, Canadian provinces, and (eventually) other countries have different rules on receipt retention, tax remittance schedules, and what counts as a "cash transaction" for AML / SAR thresholds. Hamilton's compliance recipe is wrong for Philadelphia or DC. Hardcoding Hamilton's recipe creates an audit risk for the other venues.

2. **Operational variation.** DC at peak has 3 tablets, 3 stations, possibly 3 simultaneous operators. Hamilton has 1 tablet and 1 operator at a time. The reconciliation flow that works for Hamilton (1 drop end-of-shift, 1 envelope to open, 1 manager opens it the next morning) is wrong for DC (multiple operators dropping simultaneously, multiple envelopes to count, manager may need to delegate to a back-office person who isn't the front-desk supervisor).

3. **Theft-experience adaptation.** Each venue's reconciliation rules will adapt over time as patterns emerge. Hamilton's recipe at year 1 will not be the same recipe as year 5. Hardcoding any Phase-3-implementation-time choice means re-deploying code to change the recipe later, which means change is friction-laden, which means change won't happen, which means the system will silently get worse over time as venues drift from their actual practice.

### Implementation pattern

Hamilton Settings DocType (or a new `Reconciliation Profile` DocType linked per-venue) holds the configuration:

- `float_handling: Select(Subtract / Don't Subtract)`
- `tip_rounding_rule: Select(CAD-Nickel / Penny / No-Cents)` — note: ties into existing `Currency.smallest_currency_fraction_value` discipline from PR #51 audit (CLAUDE.md → "CAD nickel rounding is site-global")
- `reveal_attempt_threshold: Int(default 3)`
- `cash_drop_frequency: Select(Per-Shift / Mid-Shift / Multiple-Per-Shift)`
- `reconciliation_cadence: Select(Daily / Weekly / Per-Drop)`
- `tolerance_pct: Percent(default 2.0)`
- `tolerance_floor_amount: Currency(default 1.00)`
- `paperwork_checklist: Table` with rows of required items

When reconciliation submits, the controller reads the venue's profile (by venue ID resolved from operator's session or current site) and applies that profile's rules — never the hardcoded Hamilton defaults.

DEC reference target: when Phase 3 starts, formalize as DEC-NNN. Today's design intent commits to "venue-configurable" without picking the exact field set; the implementer chooses the schema based on the venues actually running at that time.

## 4. Portable reconciliation session

### The decision

**Reconciliation is portable across tablets within a venue. Both the operator's count step and the manager's count step can happen on any iPad in the venue, not just the iPad where the shift was rung.**

Operator flow:
1. Operator finishes shift on Tablet A (front desk during peak).
2. Walks to the back office, picks up Tablet B (the office tablet).
3. Authenticates on Tablet B.
4. System loads the operator's open shift (all live state — drops so far, tip pulls, current sales total).
5. Operator counts cash, types declared amount on Tablet B.
6. System prints envelope label on whichever printer is paired with Tablet B.
7. Operator drops envelope in safe.
8. Front desk on Tablet A continues running normally with the next operator.

Manager flow:
1. Manager arrives next morning, picks up any tablet in the venue.
2. Authenticates.
3. System shows envelopes-awaiting-count list (filtered to whatever's in the safe but not yet reconciled).
4. Manager opens the physical envelope in front of them, picks the matching list entry on the tablet, types the count.
5. The count is recorded against that drop, no matter which tablet did the count.

### Why portable

Two driving reasons:

1. **DC-at-peak workflow.** DC's 3 tablets each run a queue. End-of-shift cash counting at the front desk blocks new check-ins because the operator and the cash count are using the same tablet. Portability lets the operator move to the office to count, freeing the front desk for the next operator.

2. **Manager-counted-elsewhere workflow.** The manager doesn't necessarily count cash at the front desk. Hamilton's small footprint means front-desk counting is fine, but DC has a back office, Philadelphia has a separate manager office, future venues may have count-rooms. Forcing reconciliation to happen on the front-desk tablet wastes those rooms' purpose-built privacy and ergonomics.

### Architectural implication

**Shift state lives in the database, not in tablet local state.** Specifically:

- Open shift records, current sales totals, in-flight cash drops, and pending reconciliations are all server-side records keyed on (operator, venue, date) — not on (tablet ID, browser session, cookie).
- Authentication on any tablet within the venue resolves the same operator's open shift.
- Tablets are stateless rendering surfaces, not stateful workstations.

This is the same architectural shape needed for **multi-tablet sync at DC** (Phase 2 requirement — when one operator on Tablet A check-ins a guest, the asset board on Tablets B and C must reflect it within seconds). The realtime broadcast channel that powers asset-board sync also serves portable reconciliation: each tablet subscribes to "venue-{venue_id}-shift-events" and re-renders when the active shift's state changes.

Practical implementation: the tablet doesn't own the shift; it queries the server for "current shift for operator X at venue Y" on every screen load. That same query works equally well from any tablet. There's no tablet-A-specific state to migrate to tablet B because there's no tablet-A-specific state.

This converges with the locking discipline (DEC-019 / `coding_standards.md` §13 three-layer locking on asset state changes): cash drop submission needs the same Redis-locked + DB-locked + version-fielded protection as asset state changes, since two operators on two tablets could theoretically race-submit drops for the same shift. Phase 1's controller doesn't have this lock yet — it's a Phase 3 add. Use the asset-board lock pattern as the reference implementation.

### Tablet-printer pairing

Each tablet needs to know which printer to send labels to. Most likely a per-tablet config (Hamilton Settings → Tablet → printer IP) rather than per-venue, since multi-tablet venues will have one printer per station. The printing endpoint accepts the originating tablet ID and routes to that tablet's paired printer. Phase 3 implementation detail; design intent: no global printer; each tablet has its own pairing.

## 5. Integrated payment (card mistype prevention)

### The problem

Standalone card terminals at Hamilton today: operator looks at the cart total, types it into the terminal, taps card, transaction completes. Three failure modes:

1. **Mistype.** Operator types $20 as $2 (or $200). Terminal accepts it. Customer's card is charged the wrong amount. The receipt printed by the terminal says $2 (or $200). The Sales Invoice in ERPNext says $20. Mismatch. Hard to detect at reconciliation because the only audit log is the operator's manually-typed amount.

2. **Wrong currency.** US-dollar customer at a CAD venue, operator forgets to switch — $20 USD gets typed as $20 CAD or vice versa.

3. **Refund / void confusion.** A void on the terminal is not always paired with a void in ERPNext, especially when the void happens hours later. The terminal's batch report may show the void; ERPNext may not. Reconciliation may flag a discrepancy that's just an unpaired refund.

The standalone-terminal failure modes are real and operator-training cannot fully fix them. Operators are tired, cash is loud, the customer is impatient, the terminal screen is small.

### The decision

**Phase 2+: integrate the card terminal with ERPNext via the Clover Connect API. ERPNext pushes the cart amount to the terminal. The operator cannot change it. The transaction completes against the system-of-record amount. The terminal's response (auth code, last 4, txn ID) is captured directly into the Sales Invoice payment line, so reconciliation has a system-side audit trail.**

### Hamilton's current terminal — confirmed Clover Flex C405 (DEC-106, 2026-05-04)

Hamilton's existing Fiserv MID is paired with a **Clover Flex C405**, confirmed 2026-05-04 in DEC-106:

- **Serial:** `C045UQ24930247`
- **Hardware revision:** 1.01
- **OS:** Android 10
- **SRED:** Enabled
- **Network:** Venue WiFi at `192.168.0.136`
- **Integration path:** Clover Connect API over WiFi

**SRED → SAQ-A PCI scope.** The terminal performs Secure Reading and Exchange of Data; the iPad / Hamilton ERP adapter receives an encrypted token only, never raw card data. This puts Hamilton in **SAQ-A PCI scope** (the simplest tier) — the expensive PCI-DSS QSA assessment + network-segmentation audit work ($5–50k/year) is avoided.

**Phase 2 integration target.** The card-adapter design pushes the cart amount to the C405 via Clover Connect API, the operator confirms on the terminal screen, the C405 performs the EMV transaction, and returns auth code + last 4 + txn ID + encrypted token to the iPad. The adapter writes those four fields into the Sales Invoice payment line. Operator never types the amount. Reconciliation has a system-side audit trail tying ERPNext's amount to the C405's batch report.

See DEC-106 in `docs/decisions_log.md` for the full spec, the SAQ-A reasoning, and the Phase-2 adapter integration plan.

### Settlement reconciliation as audit layer

A second layer of card-side audit complements the integrated-payment fix:

- Daily processor settlement report (CSV from Fiserv / Clover) is imported into ERPNext.
- A reconciliation job matches each settlement line to a Sales Invoice payment line by transaction ID.
- Unpaired lines (settled but no invoice, or invoice but no settlement) are flagged.
- Voids and refunds that show up in settlement but not ERPNext are surfaced for manager review.
- Mistypes that slipped through manual entry (Phase 1) eventually surface here when the settlement amount disagrees with the Sales Invoice amount on the same txn ID.

This audit layer runs after the operator and manager have already done their human reconciliation — it's the system catching things that escaped the human process. It does not replace blind cash control; it complements it for the card side.

### Phase 1 ships without integration

For Phase 1 launch (Hamilton, ~now), the standalone-terminal risk is accepted. Mitigation: operator training, manual receipt comparison by the manager, the Phase 3 settlement-reconciliation audit when it lands. Phase 1's Sales Invoice still records the cart amount; the operator's typed-into-terminal amount is on the printed receipt the customer takes home.

The acceptance is a known trade-off, not a denial. The settlement reconciliation will retroactively catch large mistypes against past transactions when it ships, including those from the Phase-1-with-standalone-terminal era — the data is there in Fiserv's settlement records for at least 12 months.

## 7. Browser test plan (Phase 3 implementation)

Stub-only — these are scenarios to verify when Phase 3 ships. The Phase 1 controller cannot satisfy these today (system_expected = 0 placeholder).

1. **Happy path.** Operator drops $217.30. Manager counts $217.30. POS expected $217.30. All match first attempt. Submit succeeds, variance_flag = Clean, Final Cash Amount = $217.30, posts to GL.
2. **Mismatch on attempt 1, recount, match on attempt 2.** Manager mistypes $216.30 first. System: "Doesn't match, recount." Manager recounts, types $217.30. Match. Submit succeeds, audit log records 1 failed attempt + 1 successful attempt (no reveal).
3. **Mismatch on all 3 attempts, reveal mode.** Manager types $200.00 three times in a row. After third submit, system reveals A=$217.30, B=$217.30, C=$200.00. Variance flag = "Possible Theft or Error." Manager picks "Suspected theft, escalated" from dropdown, writes a comment, types Final Cash Amount = $200.00. Submit succeeds. GL posts $200.00 as cash; $17.30 posts to "Cash variance — investigation" account.
4. **Operator opens reconciliation on Tablet A, finishes on Tablet B.** Operator starts shift close on Tablet A, walks away mid-count, picks up Tablet B, authenticates, sees the same in-progress shift, finishes count, prints label on Tablet B's paired printer. Audit log records both tablet IDs against the drop.
5. **Operator role sees blinded UI; manager role sees full UI.** Hamilton Operator role: Cash Drop form lets them type a count and submit; never shows POS expected, never shows running total, never shows other operators' drops. Hamilton Manager role: Cash Reconciliation form lets them count; pre-submit shows nothing about A or B; post-submit (or post-reveal) shows everything.
6. **Card tip rounding (CAD).** Operator types $12.67. System displays "Take $12.70 from drawer." Operator confirms. Cash Drop record stores tip_pull_amount=$12.70 and tip_pull_difference=$0.03.
7. **Required paperwork checklist blocks save.** Reconciliation form has a checklist of items the manager must check off before save. List content per venue. Save button disabled until all checked. Hamilton's list: terminal batch report attached, signed credit card slips attached. Philadelphia's list: those plus rental waiver acknowledgments.
8. **Final Cash Amount is GL truth.** Sales Invoice posted, but reveal happened and Final Cash Amount was less than POS expected. Verify: GL cash debit = Final Cash Amount (not POS expected); offsetting variance entry posts to "Cash variance — investigation" account; sales tax remittance for the period uses Final Cash Amount basis, not POS expected.
9. **Audit trail completeness post-reveal.** Reveal happened on a drop. After resolution, query the Cash Reconciliation record and verify: A, B, C all retained; all 3 manager attempts logged with timestamps; resolution dropdown choice recorded; manager comment recorded; Final Cash Amount recorded; both operator and manager IDs in the audit log; reveal_timestamp recorded; the record cannot be edited after submit (immutable).
10. **Multi-operator simultaneous drops at DC.** Three operators on three tablets each submit a drop within the same minute. Verify: each drop gets a unique Cash Drop record, no race condition on drop_number assignment, all three are independently locked + released, each is reconcilable independently. (Phase 3 / Phase 2-DC convergence test.)

## Notes for the Phase 3 implementer

- This document is design intent. Before writing code, read it twice. Then read DEC-005, DEC-021, DEC-039, DEC-041 in their canonical files. Then read the Phase 1 controller. Then write a Phase 3 implementation plan that covers all 9 browser test scenarios in §7. THEN write code.
- Do not "simplify" the blind-drop philosophy mechanics. Each rule has a cost (added complexity) but each rule has a specific failure mode it prevents. Removing any rule reopens the failure path it closed.
- Get the `system_expected` calculator right early. The Phase 1 placeholder returns 0; that's why every Phase 1 reconciliation against real data flags as "Possible Theft or Error." The first thing that must work is the real `system_expected` query: sum of cash payment lines on submitted Sales Invoices for the shift period, minus tip_pull_amount, optionally adjusted for venue-specific float-handling rule.
- Surface the audit trail prominently in the UI. Reveals are rare; when they happen, the record they create is the legal/compliance artifact. Make it visually distinct from a Clean reconciliation in list views, in PDF exports, in any reporting layer.
- When in doubt about a venue-specific rule, check the venue's profile rather than picking a default. The whole §3 architecture exists because defaults will be wrong somewhere.
