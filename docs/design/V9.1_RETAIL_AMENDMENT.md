# V9.1 Retail Amendment — Asset Board Tabs Extended to Retail SKUs

**Status:** Foundation shipped 2026-04-29. Cart UX, add-to-cart, and payment flow deliberately deferred to round 2 (see "Out of scope" below).
**Supersedes:** None. Additive amendment to V9.
**Canonical mockup:** `docs/design/V10_CANONICAL_MOCKUP.html` (asset board body unchanged from V9; retail spec lives here).
**Implementation:** `hamilton_erp/api.py`, `hamilton_erp/hamilton_erp/page/asset_board/asset_board.js`.

---

## Why this amendment exists

The Asset Board's tab framework (V9 Decision 1.2: combined config + data rule) was always architected to be per-venue configurable. V9 shipped with seven tabs covering asset categories (Lockers, Single, Double, VIP, GH Room, Waitlist, Other) plus the Watch tab. V9.1 extends the tab framework to also accept **ERPNext Item Group references** so retail SKUs (drinks, snacks, lockers' upsells) render alongside assets in the same operator UI.

This is a foundation amendment — operators see retail tabs and tiles with stock counts but **cannot yet add to cart, take payment, or process refunds**. Those interactions land in round 2 once browser testing validates the read-only retail surface.

---

## Decisions locked

These are non-negotiable for V9.1 and any later venue rollout. Reverse only with explicit Chris approval and a follow-up amendment doc.

### V9.1-D1 — SKUs live in ERPNext `Item`, not a custom DocType

Hamilton ERP stays inside ERPNext's existing inventory model. No `Hamilton SKU` DocType, no parallel data store. Every retail item is an ERPNext `Item` record.

**Why:** ERPNext Item is the universal SKU primitive. It already has `item_name`, `image`, `standard_rate` (price), `is_stock_item`, `description`, `item_group`. Building parallel storage would fork the data model and break ERPNext's reporting, valuation, tax, and integrations.

### V9.1-D2 — Inventory tracked per venue via Warehouse

Each venue has its own ERPNext Warehouse(s). Stock counts come from the standard `Bin` DocType (the per-(item, warehouse) row that ERPNext maintains automatically as Stock Entries land). The Asset Board API queries `Bin.actual_qty` filtered to the venue's default Warehouse.

**Why:** Each venue's stock policy is local. Hamilton can run "physical pull" stocking; Philadelphia can run scheduled count + restock; DC can use perpetual receipts from a vendor. The Asset Board doesn't pick a stocking method — it reads whatever ERPNext's Bin says.

### V9.1-D3 — Tabs map to `Item Group`

A retail tab represents an Item Group. The tab title is the Item Group's name (e.g. `Drink/Food`). Tiles in the tab are the Items whose `item_group` matches that Item Group.

**Why:** Item Group is ERPNext's existing taxonomy primitive. Hamilton already has 26 rooms grouped by `asset_tier` — same pattern, same operator mental model. Operators don't have to learn a new categorization scheme.

### V9.1-D4 — Per-venue `site_config.json` lists retail tabs

A new top-level key `retail_tabs` in each venue's `site_config.json` is an array of Item Group names. Each name renders as one retail tab. Empty array = no retail tabs (asset board only).

```json
{
  "anvil_venue_id": "hamilton",
  "retail_tabs": ["Drink/Food"]
}
```

**Why:** Frappe's site_config.json is already the per-venue identity layer (per the multi-venue portability research). Putting `retail_tabs` here keeps venue-specific UX choices in one file. New venues add tabs by editing their site config; no fixture edit, no migrate, no code change.

### V9.1-D5 — Tab framework is data-driven

The hardcoded `get tabs()` list in `asset_board.js` is extended to (a) keep the V9 fixed-order asset categories and (b) interleave retail tabs from `site_config.retail_tabs` between the last asset category tab and the Watch tab (which always remains far-right).

The combined visibility rule (V9 Decision 1.2) extends to retail tabs:
- Retail tab renders only when its Item Group has at least one Item with `is_active=1`.
- Empty Item Groups auto-hide just like empty asset categories.

### V9.1-D6 — Retail tile shape

Retail tiles render in the same grid as asset tiles but with different content:
- **Top-left:** Item code (e.g. `WAT-500`)
- **Top-right:** Stock count badge (e.g. `12`)
- **Body:** Item name (e.g. `Water 500ml`)
- **Footer:** Price (e.g. `$3.50`)

Tile background color reflects stock state:
- **In stock** (≥ 4 units): green (matches `--available-bg` in the V9 palette)
- **Low** (1–3 units): amber (matches `--dirty-bg` palette)
- **Out** (0 units): red (matches `--occupied-bg` palette)

Click on a retail tile **does nothing** in V9.1 (cart UX is round 2). The tile is read-only.

### V9.1-D7 — Retail tile click does NOT open an overlay

V9 asset tiles open an expand overlay on click for status actions. Retail tiles **do not** open an overlay in V9.1. The cart UX is the next round of work and will define the click behavior.

**Why:** Shipping a non-functional overlay would mislead operators in browser testing. Better to ship a clearly read-only tile so the foundation can be validated without confusion.

## API extensions

`get_asset_board_data` (the Asset Board's single read endpoint) is extended with an `items` array alongside the existing `assets` and `settings`:

```python
{
  "assets": [ ... existing asset rows ... ],
  "settings": { ... existing settings ... },
  "items": [
    {
      "name": "Water 500ml",
      "item_code": "WAT-500",
      "item_name": "Water 500ml",
      "item_group": "Drink/Food",
      "image": "/files/water.jpg",
      "standard_rate": 3.50,
      "stock": 12,
    },
    ...
  ],
  "retail_tabs": ["Drink/Food"],
}
```

`retail_tabs` echoes `site_config.retail_tabs` so the JS doesn't have to call out for it separately. `items` is fetched in two batched queries (1 query for `Item`, 1 query for `Bin`) — no N+1.

When `site_config.retail_tabs` is empty or missing, `items` is `[]` and `retail_tabs` is `[]` — a venue with no retail config gets an asset-only board with zero overhead.

## Out of scope (deferred to round 2)

These are deliberately NOT in V9.1 and will be browser-tested after this round:

- **Cart UX** — no cart modal, no running tally, no quantity stepper.
- **Add-to-cart interaction** — clicking a retail tile is a no-op in V9.1.
- **Payment flow** — no Sales Invoice creation from the Asset Board, no card terminal integration.
- **Refunds and voids** — no return path, no inventory reversal.
- **Operator-driven price overrides** — the Item's `standard_rate` is the displayed price; no per-transaction discount surface.
- **Multi-warehouse selection** — V9.1 reads from the venue's default Warehouse only. Multi-warehouse picker comes when a venue has more than one stock location.
- **Low-stock alerts** — the amber/red tile color is informational. No notification, no scheduler job, no email.

Each of these gets its own design pass after Chris validates the foundation in browser testing.

## Multi-venue rollout

This amendment is per-venue parameterized. Philadelphia, DC, Toronto, Ottawa, Dallas — each venue:
1. Edits its `site_config.json` to set `retail_tabs` (e.g. `["Drink/Food", "Merch", "Service Add-Ons"]`).
2. Has its own ERPNext Item Group(s) and Item(s) in its own database.
3. Stocks its own Warehouse via whatever stock-entry workflow that venue prefers.

Hamilton is the reference implementation. The seed patch's "Drink/Food" example is intentionally minimal so other venues can pick their own categories without inheriting Hamilton's choices.

1. `get_asset_board_data` returns `items` and `retail_tabs` keys.

4. Empty `site_config.retail_tabs` returns `items=[]` and `retail_tabs=[]`.
5. `asset_board.js` reads `data.message.items` and `data.message.retail_tabs` from the API response.
6. `asset_board.js` defines a `_render_retail_tile` method (or equivalent grouping that produces retail tile HTML).
7. `asset_board.js` filters retail tabs by their `item_group` (combined config + data rule extends).
8. `asset_board.css` defines retail-specific tile classes (`.hamilton-retail-tile`, stock-state variants).

Running `bench --site hamilton-unit-test.localhost run-tests --module hamilton_erp.test_asset_board_rendering` should report all retail tests passing.

## Browser test plan (after this PR ships)

When Chris boots `hamilton-test.localhost` and opens `/app/asset-board`, he should:

1. See a new "Drink/Food" tab between "GH Room" and "Watch" (if "Drink/Food" Item Group has Items).
2. Click "Drink/Food" — see 4 tiles (Water, Sports Drink, Protein Bar, Energy Bar) with stock badges showing 24 each.
3. Verify each tile shows item code, name, price, stock count.
4. Verify tile colors are all green (24 in stock = "in stock" range).
5. Manually create a Stock Entry that drops one item to 2 units — refresh — confirm the tile turns amber.
6. Drop to 0 — confirm tile turns red.
7. Click a retail tile — confirm nothing happens (no overlay, no error).
8. Open the existing asset tabs (Lockers, Single, etc.) — confirm asset board still works exactly as it did pre-V9.1.

Once all 8 verifications pass, the foundation is locked and round 2 (cart UX) can begin.
