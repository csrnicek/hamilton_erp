# V9.1 Retail Amendment — Asset Board Tabs Extended to Retail SKUs

**Status:** Foundation shipped 2026-04-29. Cart UX, add-to-cart, and payment flow deliberately deferred to round 2 (see "Out of scope" below).
**Supersedes:** None. Additive amendment to V9.
**Canonical mockup:** `docs/design/V10_CANONICAL_MOCKUP.html` (asset board body unchanged from V9; retail spec lives here).
**Implementation:** `hamilton_erp/api.py`, `hamilton_erp/hamilton_erp/page/asset_board/asset_board.js`.

---

## Why this amendment exists

The Asset Board's tab framework (V9 Decision 1.2: combined config + data rule) was always architected to be per-venue configurable. The canonical V9 tab taxonomy is **6 asset-category tabs** (Lockers, Single, Double, VIP, Waitlist, Other) plus the Watch tab — see `docs/decisions_log.md` §1.1 "Seven-tab structure (fixed order)" and §7 "Category is fixed, subtypes are not." Subtypes within a category (e.g. "GH Room", "Glory Hole", "TV") are **venue-specific data, NOT hardcoded tabs** — Hamilton happens to have GH Room as a Single subtype, but Philadelphia / DC / Dallas may have different subtypes within the same 6-tab framework. V9.1 extends the tab framework to also accept **ERPNext Item Group references** so retail SKUs (drinks, snacks, lockers' upsells) render alongside assets in the same operator UI; the asset-category tab set itself remains the canonical 6 + Watch.

This is a foundation amendment — operators see retail tabs and tiles with stock counts but **cannot yet add to cart, take payment, or process refunds**. Those interactions land in round 2 once browser testing validates the read-only retail surface.

---

## Decisions locked

These are non-negotiable for V9.1 and any later venue rollout. Reverse only with explicit Chris approval and a follow-up amendment doc.

### V9.1-D1 — SKUs live in ERPNext `Item`, not a custom DocType

Hamilton ERP stays inside ERPNext's existing inventory model. No `Hamilton SKU` DocType, no parallel data store. Every retail item is an ERPNext `Item` record.

**Why:** ERPNext Item is the universal SKU primitive. It already has `item_name`, `image`, `standard_rate` (price), `is_stock_item`, `description`, `item_group`. Building parallel storage would fork the data model and break ERPNext's reporting, valuation, tax, and integrations.

### V9.1-D2 — Inventory tracked per venue via Warehouse

Each venue has its own ERPNext Warehouse(s). Stock counts come from the standard `Bin` DocType (the per-(item, warehouse) row that ERPNext maintains automatically as Stock Entries land). The Asset Board API queries `Bin.actual_qty` filtered to the venue's default Warehouse.

**Why:** Each venue's stock policy is local. Hamilton can run "physical pull" stocking; Philadelphia can run scheduled count + restock; DC can use perpetual receipts from a vendor. The Asset Board doesn't pick a stocking method — it reads whatever ERPNext's Bin says.

### V9.1-D3 — Tabs map to `Item Group`

A retail tab represents an Item Group. The tab title is the Item Group's name (e.g. `Drink/Food`). Tiles in the tab are the Items whose `item_group` matches that Item Group.

**Why:** Item Group is ERPNext's existing taxonomy primitive. Hamilton already has 26 rooms grouped by `asset_tier` — same pattern, same operator mental model. Operators don't have to learn a new categorization scheme.

### V9.1-D4 — Per-venue `site_config.json` lists retail tabs

A new top-level key `retail_tabs` in each venue's `site_config.json` is an array of Item Group names. Each name renders as one retail tab. Empty array = no retail tabs (asset board only).

```json
{
  "anvil_venue_id": "hamilton",
  "retail_tabs": ["Drink/Food"]
}
```

**Why:** Frappe's site_config.json is already the per-venue identity layer (per the multi-venue portability research). Putting `retail_tabs` here keeps venue-specific UX choices in one file. New venues add tabs by editing their site config; no fixture edit, no migrate, no code change.

### V9.1-D5 — Tab framework is data-driven

The hardcoded `get tabs()` list in `asset_board.js` is extended to (a) keep the V9 fixed-order asset categories and (b) interleave retail tabs from `site_config.retail_tabs` between the last asset category tab and the Watch tab (which always remains far-right).

The combined visibility rule (V9 Decision 1.2) extends to retail tabs:
- Retail tab renders only when its Item Group has at least one Item with `is_active=1`.
- Empty Item Groups auto-hide just like empty asset categories.

### V9.1-D6 — Retail tile shape

Retail tiles render in the same grid as asset tiles but with different content:
- **Top-left:** Item code (e.g. `WAT-500`)
- **Top-right:** Stock count badge (e.g. `12`)
- **Body:** Item name (e.g. `Water 500ml`)
- **Footer:** Price (e.g. `$3.50`)

Tile background color reflects stock state:
- **In stock** (≥ 4 units): green (matches `--available-bg` in the V9 palette)
- **Low** (1–3 units): amber (matches `--dirty-bg` palette)
- **Out** (0 units): red (matches `--occupied-bg` palette)

Click on a retail tile **does nothing** in V9.1 (cart UX is round 2). The tile is read-only.

### V9.1-D7 — Retail tile click does NOT open an overlay (SUPERSEDED 2026-04-30)

V9 asset tiles open an expand overlay on click for status actions. Retail tiles do not open an overlay; instead, V9.1-D9 (below) defines the click behavior.

**Original reasoning (preserved for history):** Shipping a non-functional overlay would mislead operators in browser testing.

**Superseded by:** V9.1-D9 (tap retail tile = add to cart). The "click does nothing" wording was for the V9.1 foundation PR; round 2 (Phase 2 cart UX) gives the click a real behavior.

## Phase 2 cart UX (round 2) — V9.1-D8..D15

Shipped in two PRs on 2026-04-30:
1. **PR #49 (morning) — UX-only stub.** Cart state machine, drawer rendering, qty controls, HST math, and cash payment modal all work end-to-end in the browser; Sales Invoice creation is a no-op toast.
2. **Follow-up PR (afternoon) — accounting prereqs + Sales Invoice wiring.** QBO-mirrored accounts (4260 Beverage, 4210 Food, GST/HST Payable), Hamilton warehouse + cost center, Ontario HST 13% tax template, POS Profile "Hamilton Front Desk", and the `submit_retail_sale` API. The cart Confirm now creates a real POS Sales Invoice that decrements stock on submit.

### V9.1-D8 — Cart drawer at the bottom (slide-up, dark, fixed)

When the cart has at least one line, a drawer is fixed-positioned at the bottom of the asset board. Two states:

- **Collapsed (default):** one row showing item count + grand total + chevron. Tapping the row expands.
- **Expanded:** lines with qty controls (− / + buttons), subtotal / HST / total breakdown, and two action buttons (Clear, Cash payment).

When the cart is empty, the drawer DOM is cleared (no empty-state shown). The drawer is also visible across all tabs — switching from a retail tab to an asset tab keeps the cart visible so the operator can see what's pending while glancing at the asset board.

**Why:** Bottom drawer is the POS-conventional placement; works on iPad in landscape; doesn't compete with the right-side header for space. Fixed position survives scrolling within a tab.

### V9.1-D9 — Tap retail tile = add 1 to cart (supersedes V9.1-D7)

First tap on a retail tile creates a cart line with `qty=1`. Each subsequent tap increments the existing line's qty by 1. The tile shows a green "In cart: N" pill in the bottom-right when the line exists.

**Out-of-stock guard:** tapping a tile with `stock <= 0` shows a red toast "Out of stock" and does NOT add the line. Operators can't create a cart line for an item they can't sell.

### V9.1-D10 — Qty controls in the drawer (− / + buttons)

Each cart line has − and + buttons. The − button decrements by 1; reaching `qty=0` removes the line. There is no numeric input field (Phase 2 first cut keeps the keyboard out of the way for tablet operators).

### V9.1-D11 — HST 13% applied flat to subtotal

`HST_RATE = 0.13` is a top-level JS constant. The drawer shows three lines: Subtotal, HST 13%, Total. HST is computed as `Math.round(subtotal × 0.13 × 100) / 100` (banker's-rounding-adjacent; rounded half-up at 2 decimals).

When a venue outside Ontario onboards (e.g. Philadelphia at 6% PA sales tax), this becomes a per-venue config in `site_config.json`. For now it's a hardcoded constant.

### V9.1-D12 — Cash payment, single tender, modal — Confirm creates a POS Sales Invoice

Tapping "Cash payment" opens a modal showing the total + a numeric input for cash received. As the operator types, the modal computes change in real time; the Confirm button enables only when received >= total.

**Tapping Confirm calls `hamilton_erp.api.submit_retail_sale`** which creates a POS Sales Invoice (`is_pos=1, update_stock=1`) against the "Hamilton Front Desk" POS Profile. On success, the cart clears, a green toast shows the invoice ID + change owed, and the asset board refreshes so retail tile stock counts update.

On error (insufficient stock, accounting misconfiguration, etc.), the cart is preserved so the operator can retry without re-entering items.

**Why this is a real wire and not a stub:** Hamilton's accountant decided 2026-04-30 to mirror QBO names exactly — `GST/HST Payable`, `4260 Beverage`, `4210 Food`. With the names locked, the seed is straightforward and reversible (drop the Company on a fresh install to start over).

### V9.1-D15 — Accounting prereqs seeded by `_ensure_hamilton_accounting`

Per the QBO mirror decision (locked by Chris's accountant 2026-04-30), the install/migrate seed creates:

- **Company:** `Club Hamilton` (abbr `CH`) with Standard CoA, currency CAD, country Canada — created only if no Hamilton-named company exists. Pin via `bench --site SITE set-config hamilton_company "Existing Co"` for production sites that already have a real company.
- **Warehouse:** `Hamilton - {abbr}` under company root warehouse.
- **Cost Center:** `Hamilton - {abbr}` under company root cost center.
- **Income accounts (under Indirect Income):** `4260 Beverage - {abbr}`, `4210 Food - {abbr}`.
- **Tax account (under Duties and Taxes):** `GST/HST Payable - {abbr}` with `account_type=Tax, tax_rate=13`.
- **Sales Taxes Template:** `Ontario HST 13% - {abbr}` referencing the GST/HST Payable account.
- **Mode of Payment Account:** Cash → `Cash - {abbr}` for the Hamilton company.
- **Stock Settings:** `default_warehouse = Hamilton - {abbr}` (only if currently unset).
- **POS Profile:** `Hamilton Front Desk` — operator-invisible, referenced by name in `submit_retail_sale`. Carries the warehouse, cost center, currency, tax template, and Cash payment method.
- **Item Defaults:** wired on the 4 seeded retail items (WAT-500, GAT-500 → 4260 Beverage; BAR-PROT, BAR-ENRG → 4210 Food).

The seed is idempotent — every step uses `frappe.db.exists()` guards. Re-runs on partial state heal forward without crashing.

### V9.1-D13 — Cart state is per-session, JS-only, no persistence

The cart lives in `this.cart` array on the AssetBoard instance. Lost on page reload by design. Operators ring up sales atomically; an abandoned cart should not survive into the next operator's session. Phase 3 may revisit if a documented use case emerges (e.g. operator handoff mid-transaction).

### V9.1-D14 — Out of scope for this round

- **Card payments** — Phase 2 next iteration (Helcim/Stripe adapter scaffolding, capture `merchant_transaction_id` per sale). Tap-to-pay is treated as the "Card" Mode of Payment, not a separate method. See `docs/inbox.md` 2026-04-30 Phase 2 hardware backlog for full spec.
- **Multiple active merchants + manual failover** — Phase 2 next iteration. Per DEC-064 (locked 2026-05-01), every venue must have a primary AND backup processor pre-approved and integration-tested; system must support add/swap without code changes. Per-venue config in `site_config.json`. (Hamilton itself is standard-classified per DEC-062, but any merchant can be terminated for chargeback drift / underwriter changes / M&A — backup architecture is general resilience, not classification-specific.)
- **Receipt printing (Epson TM-T20III)** — Phase 2 next iteration. Auto-prints every transaction; receipt doubles as physical asset board control token (paper on key hook = occupied). Pattern mirrors Brother label printer (DEC-011).
- **Digital receipt (email/SMS)** — Phase 2 next iteration. Offered alongside paper, not as a replacement.
- **Split tender** (cash + card) — Phase 3+.
- **Discounts / comps at the cart line** — out of scope; not a regular operator path.
- **Admission items in cart** — the existing `assign_asset_to_session` Phase 2 stub remains untouched. Retail items only.
- **Loyalty / member pricing** — Phase 4 (membership system isn't built).

## API extensions

`get_asset_board_data` (the Asset Board's single read endpoint) is extended with an `items` array alongside the existing `assets` and `settings`:

```python
{
  "assets": [ ... existing asset rows ... ],
  "settings": { ... existing settings ... },
  "items": [
    {
      "name": "Water 500ml",
      "item_code": "WAT-500",
      "item_name": "Water 500ml",
      "item_group": "Drink/Food",
      "image": "/files/water.jpg",
      "standard_rate": 3.50,
      "stock": 12,
    },
    ...
  ],
  "retail_tabs": ["Drink/Food"],
}
```

`retail_tabs` echoes `site_config.retail_tabs` so the JS doesn't have to call out for it separately. `items` is fetched in two batched queries (1 query for `Item`, 1 query for `Bin`) — no N+1.

When `site_config.retail_tabs` is empty or missing, `items` is `[]` and `retail_tabs` is `[]` — a venue with no retail config gets an asset-only board with zero overhead.

## Multi-venue rollout

This amendment is per-venue parameterized. Philadelphia, DC, Toronto, Ottawa, Dallas — each venue:
1. Edits its `site_config.json` to set `retail_tabs` (e.g. `["Drink/Food", "Merch", "Service Add-Ons"]`).
2. Has its own ERPNext Item Group(s) and Item(s) in its own database.
3. Stocks its own Warehouse via whatever stock-entry workflow that venue prefers.

Hamilton is the reference implementation. The seed patch's "Drink/Food" example is intentionally minimal so other venues can pick their own categories without inheriting Hamilton's choices.
